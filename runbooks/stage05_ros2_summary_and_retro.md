# Stage 05 (MORAI↔ROS2) — 정리 & 회고

2026-07-03 세션. MORAI SIM 26.R1.H3 의 데이터를 host ROS2 Humble 로 받는 작업. AVS-007 관련.

---

## PART 1. 쉬운 정리 — 지금 상황

### 목표
MORAI SIM 이 만드는 Ego 상태·센서 데이터를 host 의 ROS2(Humble) 토픽으로 받아서, 외부 알고리즘/
Autoware 가 쓰고 sim/wall 시간 offset 을 재는 것.

### 두 가지 연결 방법과 결과
| 방법 | 원리 | 결과 | 원인 |
|---|---|---|---|
| **① ROS2 Native** | SIM 내장 ros2cs 가 DDS 로 직접 발행 | ❌ SIM 이 켜자마자 죽음 | 내장 ros2cs 가 **2023년 빌드(standalone=0)** 라 host ROS2 를 빌려쓰는데, host 는 **2026 버전** → `std::bad_cast`. **host 버전을 2023 으로 맞춰봐도 동일 실패**(실험으로 확인) → host 버전 문제 아님, MORAI 플러그인 로딩 문제 |
| **② rosbridge** | SIM 이 WebSocket 으로 보내고 host 가 ROS2 토픽으로 변환 | ⚠️ 토픽은 생기는데 **데이터가 안 흐름** | SIM 이 **ROS1 헤더(`header.seq`)** 로 보내는데 ROS2 에는 `seq` 필드가 없어서 rosbridge 가 거부. **딱 이 필드 하나가 문제 → 고칠 수 있음(미시도)** |

### 한 줄 결론
- **Native**: MORAI 지원 필요(그들이 만든 2023 ros2cs 가 최신 host 와 안 맞음). 우리가 못 고침.
- **rosbridge**: `header.seq` 하나 때문 → **우리 쪽 rosbridge 패치로 살릴 가능성 높음. 다음에 테스트.**

  확보 → offset 측정 → Stage 05 통과 시도. (§ AVS-007 report / run_rosbridge.sh)

## PART 2. 회고 — 왜 처음에 분석이 어려웠나, 개선점

### 무엇이 시간을 잡아먹었나 (방향 전환의 연속)
초기 가정(ROS2 native + CycloneDDS) → 실제로는 rosbridge 도 봐야 했고 → RMW 는 cyclonedds 가 아니라
fastrtps → "ROS 를 격리해야 한다"(오판) → 사실은 반대로 "ROS 를 소싱해야 한다" → "버전 정합"(실험 후
반증) → rosbridge 로 전환 → "근본 포맷 불일치"(오판) → 실제로는 `header.seq` 한 필드. **여러 번 뒤집혔고
매번 SIM 재기동/재연결이 필요해 느렸다.**

### 근본 원인 5가지 + 개선점

1. **실물 바이너리보다 외부 문서/가정을 먼저 믿었다.**
   - 처음에 MORAI 웹문서 기반으로 "CycloneDDS" 로 갔지만, SIM 실물은 fastrtps 였다. SIM 설치본의
     `metadata_ros2cs.xml`·Plugins·ldd 를 **먼저** 봤다면 방향이 단번에 잡혔다.
   - **개선**: 벤더 연동은 **실물 아티팩트(메타데이터/.so 의존성/번들 라이브러리/버전)를 가장 먼저** 조사하고
     동작 모델을 글로 확정한 뒤 손대기. 문서는 버전이 어긋날 수 있다(실제로 어긋났다).

2. **전체를 특성화하기 전에 매 단서마다 즉시 조치했다.**
   - "격리하자" 처럼 성급히 바꿨다가 `standalone=0` 을 발견하고 정반대로 되돌렸다. 매 변경이 SIM
     재기동을 요구해 비용이 컸다.
   - **개선**: 미지의 서브시스템은 **읽기 전용 특성화 패스(로그/ldd/메타/`/proc`)를 한 번에** 끝내 모델을
     세운 뒤, 값비싼(사람+GUI) 테스트를 최소 횟수로.

3. **초반에 넣은 `.bashrc` 자동 소싱이 혼란을 키웠다.**
   - 편의로 넣은 ROS2 자동 소싱이 런처가 spawn 하는 SIM 셸에 ROS 를 재주입해, "격리" 삽질을 유발.
   - **개선**: 전역 환경(`.bashrc`) 변경은 **자기 런타임을 내장한 벤더 GUI 앱**에 영향 줄 수 있으니 지양.
     환경은 스코프 한정(필요 터미널에서만 `rosenv`/`env.sh`).

4. **로그 잘림(truncation)이 결정적 단서(`header.seq`)를 오래 숨겼다.**
   - `/rosout` 로 본 rosbridge 에러가 잘려 "필드 불일치"까지만 보였다. **사용자가 rosbridge 터미널 원문을
     붙여준 마지막 순간**에야 `header.seq` 가 드러났다. 그 전까진 "근본 불일치"로 과잉 결론.
   - **개선**: 로그가 잘리면 **중계본(/rosout)이 아니라 프로세스 원본 stderr/터미널**을 확보. 실패
     프로세스의 실제 콘솔 출력을 **일찍** 요청.

5. **테스트 루프가 느리고 사람 손(SIM GUI)에 묶여 있었다.**
   - 가설 검증마다 SIM 실행→지도→Network Settings→Connect→Play 가 필요해 반복이 느리고 상태가 불투명.
   - **개선**: **SIM 없이 재현 가능한 것**(정적 ldd/dlopen, rclpy 재현)을 먼저. 이번에 잘한 부분이기도 하다
     (버전정합 반증을 rclpy 로 확정, `/proc/environ` 캡처).

### 잘한 점(유지)
- 버전정합 실험(#2)을 스냅샷 deb + rclpy 대조로 **깔끔히 반증** → 무의미한 전체 다운그레이드 회피.
- 정적 분석(metadata/ldd/`/proc`) 이 일단 적용되자 원인 규명이 빨랐다.
- 모든 단계를 이슈/증거/리포트로 기록 → 작업이 유실되지 않음.

### 한 줄 교훈
> **"벤더 실물부터, 읽기 전용으로 전부 특성화한 뒤, 원본 로그로, SIM 없이 재현" — 이 순서였다면
> 방향 전환과 SIM 재기동을 크게 줄였을 것.**
